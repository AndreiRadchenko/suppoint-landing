# SUPPOINT Landing Constitution

## Core Principles

### I. Conventional Vite Vanilla-JS Structure
The codebase MUST follow the established Vite multi-page layout: HTML entries at the
repository root, application code under `src/js/`, styles under `src/styles/`,
Handlebars partials under `src/partials/`, and static assets under `src/assets/`.
Build-time scripts live in `scripts/` at root; `public/` holds only un-hashed static
files (e.g. `CNAME`). This convention keeps the project navigable and consistent with
standard Vite starter templates.

### II. Design System Fidelity (Liquid Glass)
All UI MUST conform to the Liquid Glass design system defined in `DESIGN.md`: aquatic
blue palette, Hanken Grotesk headlines + Plus Jakarta Sans body, glassmorphic surfaces
with backdrop blur, generous spacing on an 8px base unit, and pill/rounded shapes.
Sunny Orange is reserved strictly for primary CTAs and the logo. New components MUST
reuse existing design tokens rather than introduce ad-hoc colors, fonts, or radii.

### III. No Behavior Change Without Intent
Refactors and restructuring (e.g. moving files) MUST be behavior-preserving unless the
change is explicitly requested. Path, import, and asset-reference updates that accompany
a move MUST be verified with `npm run build` before completion. The generated `price.js`
and `dist/` build output MUST NOT be hand-edited or committed.

### IV. Internationalization by Convention
User-facing copy MUST be externalized through the client-side i18n system: elements are
marked with `data-i18n` attributes and translations live in `src/js/translations.js`
(Ukrainian default, English secondary, persisted in `localStorage`). Hard-coded visible
strings in HTML/JS are non-compliant. Route/map coordinates MUST come from translation
keys, not literals.

### V. Verifiable Via Build, Not a Test Suite
The project has no automated test or lint setup. Correctness MUST therefore be verified
by running `npm run build` (which runs the `prebuild` price fetch then `vite build`)
and `npm run dev` for manual review. Any change that breaks the production build is a
blocking defect. Formatting conventions (Prettier: no semicolons, single quotes, 2-space
indent, `printWidth: 100`) MUST be applied via `npm run format` before commit.

## Project Structure

The canonical layout is:

```text
index.html, about.html, contact.html   # Vite entry points (root)
src/
  js/            # main.js, i18n.js, translations.js, price.js (generated), components/
  styles/        # style.css (Tailwind v4), fonts.css
  partials/      # Handlebars partials (head, nav, footer, sections)
  assets/        # images/ and fonts/ (Vite-hashed on build)
scripts/         # update-prices.js (regenerates src/js/price.js)
public/          # CNAME and other un-hashed static files
```

`price.js` is AUTO-GENERATED by `npm run update-prices` and MUST NOT be edited manually.
`dist/` is git-ignored build output and MUST NOT be committed.

## Workflow & Quality Gates

- Every change MUST pass `npm run build` before being considered complete.
- Code MUST be formatted with `npm run format` (Prettier) prior to commit.
- i18n strings MUST be added to `translations.js` for both `uk` and `en`.
- New sections/partials MUST follow existing naming and the Liquid Glass tokens in `DESIGN.md`.
- Commits SHOULD be small and scoped; the repo uses conventional-ish messages
  (`refactor:`, `fix:`, `chore:`, `docs:`).

## Governance

This constitution supersedes ad-hoc practices. It is the source of truth for structure,
design fidelity, and verification discipline.

- **Amendments**: Any change to a principle or section requires updating this file,
  incrementing the version per semantic versioning, and recording the change in the
  Sync Impact Report comment at the top of the file.
- **Versioning**: MAJOR = removal/redefinition of a principle; MINOR = new principle or
  materially expanded guidance; PATCH = clarifications or wording fixes.
- **Compliance review**: Every PR MUST verify build success, formatting, and design/i18n
  conformance. Runtime development guidance lives in `AGENTS.md` and `DESIGN.md`; this
  constitution takes precedence when they conflict.
